# 计划文档技能

这个技能管理代码实现前的规划流程，确保开发计划经过彻底的审查和批准。

## 使用时机

**必须在以下情况调用此技能：**

- 用户要求编写或生成代码
- 即将实现新功能或进行重大更改
- 任何需要在实现前进行规划的开发任务

## 工作流程

### 1. 计划目录创建

在编写任何代码之前，AI必须：

- 创建与预期代码目录结构对应的计划目录
- 示例：如果代码将放在 `src/components/Button.tsx`，则在 `plans/src/components/Button.md` 创建计划

### 2. 计划文档生成

每个计划文档应包含：

- **目标**：明确描述将实现的内容
- **方法**：技术方法和设计决策
- **要创建/修改的文件**：受影响文件的列表
- **实现步骤**：详细的分步计划
- **依赖项**：所需的库、API或外部资源
- **测试策略**：如何测试实现

### 3. 用户审查

生成计划文档后：

- **等待**用户审查计划
- 在用户明确批准前，不得进行代码生成
- 允许用户标注或修改计划文档

### 4. 计划调整

如果用户提供反馈或要求更改：

- 根据用户输入修改计划文档、用户可能直接在计划文档中输入要调整的部分（这种情况用户只会在计划文档最后加一个‘## 调整’的标题，后面会有具体的调整内容）、你每次都需要检查计划文档中是否有此标题，若有则根据标题后面的内容调整计划并在调整后将调整部分的提示内容移除
- 重新提交审查
- 重复调整周期，直到用户满意

### 5. 代码生成

只有在用户明确批准计划后：

- 按照批准的计划进行代码实现
- 保持与计划结构和决策的一致性

## 计划文档模板

```markdown
# 计划：[组件/功能名称]

## 目标

[明确描述将构建的内容]

## 方法

[技术方法和设计决策]

## 文件结构

- `path/to/file1.ext`：[用途]
- `path/to/file2.ext`：[用途]

## 实现步骤

1. [步骤1]
2. [步骤2]
3. [步骤3]

## 依赖项

- [库/包]：[版本/用途]
- [外部API]：[详情]

## 测试策略

- [测试方法1]
- [测试方法2]
```

## 重要规则

- **绝不要**在未创建并获得计划批准的情况下生成代码
- **始终**创建与预期代码结构对应的计划目录
- **等待**用户明确批准后再开始实现
- **尊重**用户对计划文档的标注和修改
- **迭代**计划，直到用户满意后再编码
